# ID Aura 规范库更新与自动发布指南

> 适用对象：**负责「新建/维护规范库」任务的团队**
> 目标：编辑规范内容 → 自动发布到 GitHub → 用户打开 ID Aura 即可拿到最新规范，**无需重装软件**。

---

## 一、规范库是什么，它怎么到达用户

ID Aura 左侧「规范库」面板里所有餐饮 / 酒店 / 家装 / 建筑 / 景观的规范条目，数据来自一个放在 GitHub 上的 JSON 文件。软件每次启动时自动联网拉取最新版本。

```
[你编辑 spec.json]
       │  commit + push 到 GitHub (main 分支)
       ▼
[仓库根目录 spec.json]  ──jsDelivr CDN──▶  [用户电脑上的 ID Aura]
       │                                        （启动时自动拉取，线上即权威、直接覆盖本地）
       └──GitHub raw 兜底源──▶  [同上]
```

关键点：

- **用户实际下载的文件** = 仓库根目录的 `spec.json`（不是 `public/`，也不是 `dist/`）。
- 默认拉取地址：`https://cdn.jsdelivr.net/gh/chengcheng067/Id-aura@main/spec.json`
- 兜底地址：`https://raw.githubusercontent.com/chengcheng067/Id-aura/main/spec.json`
- 软件启动即把线上数据当作**权威**：只要能联网取到品类数据，就**直接覆盖本地缓存**，不再要求远程版本号大于本地。所以你改完内容并提交后，用户下次打开即生效——`version` 字段仅用于展示与离线缓存标识（**建议每次发布仍升一档，便于回溯**，但不再是覆盖的前置条件）。

## 二、仓库里和规范的角色清单

| 文件 | 角色 | 团队要碰吗 |
|---|---|---|
| `spec.json`（仓库根） | **线上源**，用户实际下载的规范 | ✅ 主要编辑对象 |
| `src/data/specs.bundled.js` | EXE 内置离线兜底（自动生成，勿手改） | ❌ 仅在「全量同步」时由脚本处理 |
| `scripts/gen-spec.mjs` | 从 bundled 生成根 `spec.json` | ❌ 发版流程用 |
| `scripts/publish-spec.mjs` | **自动上传**：校验+提交+推送+刷缓存 | ✅ 发布时运行 |
| `scripts/json-to-csv.mjs` | 导出规范为 Excel 可编辑的 CSV | ✅ 批量编辑用 |
| `scripts/specs-csv-to-json.mjs` | 把编辑好的 CSV 导回 `spec.json` | ✅ 批量编辑用 |
| `scripts/beautify-spec.mjs` | 把 `spec.json` 变成可读多行格式 | ✅ 手工编辑前用 |

> 软件侧还有个「设置 → 规范」Tab，那是**用户**用来查看版本/来源、或临时切换规范源的，与你们发布无关。

## 三、数据格式（必读，写错会导致发布被拦截）

`spec.json` 顶层结构：

```json
{
  "version": "1.0.0",
  "data": {
    "hotpot": {
      "name": "火锅店",
      "category": "室内设计规范",
      "subCategory": "餐饮工装",
      "icon": "🍲",
      "description": "火锅店设计需重点考虑排烟通风、电磁炉供电、食材动线",
      "sections": [
        {
          "title": "动线与通道",
          "items": [
            { "label": "主通道宽度", "value": "≥ 1200mm", "note": "消防疏散要求，至少1.2米", "priority": "high" }
          ]
        }
      ]
    }
  }
}
```

### 字段规则

| 字段 | 位置 | 说明 / 取值 |
|---|---|---|
| `version` | 顶层 | **必须**是 `x.y.z` 格式，且每次发布要比上一次大。 |
| `data` | 顶层 | 对象，**键 = 品类 ID**（英文小写，如 `hotpot`、`hotel-guestroom`）。 |
| `name` | 品类 | 中文显示名，如「火锅店」。 |
| `category` | 品类 | 三大主类之一：`建筑设计规范` / `室内设计规范` / `景观设计规范`。 |
| `subCategory` | 品类 | 二级分组，现有取值见下。 |
| `icon` | 品类 | 一个 emoji。 |
| `description` | 品类 | 一句话说明。 |
| `sections[].title` | 分组 | 该品类下的小节标题，如「动线与通道」。 |
| `items[].label` | 条目 | 规范项名称，如「主通道宽度」。 |
| `items[].value` | 条目 | 标准值，如「≥ 1200mm」。 |
| `items[].note` | 条目 | 补充说明。 |
| `items[].priority` | 条目 | **只能是 `high` / `medium` / `low` 三者之一**（小写英文），决定界面左侧色条颜色。 |

### 现有 `subCategory` 取值（新增品类时尽量复用，保持归类一致）

- 餐饮工装、酒店民宿、家装
- 办公建筑、商业建筑、教育建筑、医疗建筑、住宅建筑、工业建筑、交通建筑
- 公园景观、商业景观、居住区景观、屋顶花园、水景设计、铺装设计、植物配置

> 当前规模：**45 个品类 / 1087 条规范**。新增品类时 `categoryId` 用全新英文 id 即可，软件会自动出现。

### 顶层 `app` 节点（宣布「软件本体有新版本」用）

`spec.json` 除了 `version` + `data`，还可以带一个**顶层 `app` 节点**。它和「规范数据版本」完全独立——专门用来告诉**已经安装旧版 EXE 的用户**：「有新版本可下载了」。软件启动时会解析这个节点，对比用户本地版本，发现更新就在工具栏弹提示。

```json
{
  "version": "1.1.0",
  "data": { "...": "（规范数据，照旧）" },
  "app": {
    "latestVersion": "2.10.0",
    "minVersion": "2.9.0",
    "critical": false,
    "downloadUrl": "https://github.com/chengcheng067/Id-aura/releases/latest",
    "releaseNotes": "• 软件自动自检更新\n• 规范库热更新",
    "publishDate": "2026-07-14"
  }
}
```

`app` 节点字段：

| 字段 | 说明 |
|---|---|
| `latestVersion` | **当前可下载的最新软件版本**（如 `2.10.0`）。用户本地版本比它旧 → 提示更新。 |
| `minVersion` | 最低兼容版本（预留，当前未强制）。 |
| `critical` | `true` 时隐藏「稍后再说 / 忽略此版本」，逼用户必须去下载。 |
| `downloadUrl` | 点「去下载」打开的地址，默认 GitHub Releases 最新页。 |
| `releaseNotes` | 更新说明，`\n` 分行展示在弹窗里。 |
| `publishDate` | 发布日期，仅供参考。 |

> ⚠️ **发布纪律**：`app.latestVersion` 必须和「实际已发布到 GitHub Releases 的版本」一致。**不要在 EXE 还没发布时就提前改大它**——否则用户点了「去下载」会跳到尚不存在的页面。等构建完 EXE 并发布后，再把 `app.latestVersion` 改成新版本号并提交 `spec.json` 即可。
>
> 好消息：`app` 节点与「规范数据版本 `version`」互不影响。你日常改规范升的是 `version`；发软件新版时升的是 `app.latestVersion`。两者可独立提交。
>
> 另：`scripts/gen-spec.mjs` 在重新生成 `spec.json` 时会**保留**已有的 `app` 节点，规范数据重新生成不会误清发布版本信息。

## 四、三种更新方式（选一种）

### 方式 A：直接改 `spec.json`（★ 推荐，日常热更新）

适合少量增改。

```bash
# 1) 变成可读的多行格式（只需在仓库初次整理时跑一次）
node scripts/beautify-spec.mjs

# 2) 用任意编辑器（VS Code / 记事本）打开 spec.json，增改内容
#    注意：把顶层的 "version": "1.0.0" 改成更大的值，如 "1.0.1"

# 3) 发布（自动校验 + 提交 + 推送 + 刷缓存）
node scripts/publish-spec.mjs --msg="chore(spec): 火锅店增补排烟规范 v1.0.1"
```

### 方式 B：用 Excel/CSV 批量编辑（适合大批量新增）

适合一次加很多条目、或不熟悉 JSON 的同事。

```bash
# 1) 导出为 CSV（生成 specs.csv，可用 Excel 打开）
node scripts/json-to-csv.mjs

# 2) 用 Excel 打开 specs.csv，按列编辑/新增行
#    列：categoryId, category, subCategory, name, icon, description,
#        sectionTitle, label, value, note, priority
#    注意 priority 只能填 high / medium / low

# 3) 存回 CSV 后，导回 spec.json
node scripts/specs-csv-to-json.mjs specs.csv --version=1.1.0

# 4) 发布
node scripts/publish-spec.mjs --msg="chore(spec): 批量新增酒店民宿规范 v1.1.0"
```

CSV 示例（第一行是表头，不要删）：

```csv
categoryId,category,subCategory,name,icon,description,sectionTitle,label,value,note,priority
hotpot,室内设计规范,餐饮工装,火锅店,🍲,火锅店设计需重点考虑排烟通风,动线与通道,主通道宽度,≥ 1200mm,消防疏散要求，至少1.2米,high
newcat,室内设计规范,餐饮工装,新品类,🆕,说明文字,小节标题,条目名,标准值,备注,medium
```

### 方式 C：同时更新 EXE 离线兜底（仅发版时做）

`spec.json` 是线上热更新源；EXE 安装包里还内置了一份离线兜底 `specs.bundled.js`（断网/重装时用）。**日常维护不需要动它**。只有当你希望「即使离线，新装的用户也带最新规范」时，才在打安装包前做一次全量同步：

```bash
# 先确保根 spec.json 是最新的，然后反转生成 bundled 兜底（开发者执行）
node scripts/gen-spec.mjs
# 随后走正常的 EXE 打包流程（这条与团队日常无关，交给发版人）
```

## 五、发布前会自动检查的校验（不用手记，脚本会拦）

`publish-spec.mjs` 在提交前会逐一核对，**不通过就中止并告诉你哪条错了**：

1. `spec.json` 是合法 JSON；
2. `version` 为 `x.y.z` 且**比上一次提交更大**；
3. 每个品类都有 `name / category / subCategory / icon / description`；
4. 每个 `sections` 有 `title`，每个 `items` 有 `label / value / note`；
5. 每个 `priority` 只能是 `high / medium / low`。

演练（只检查、不动文件、不推送）：

```bash
node scripts/publish-spec.mjs --dry-run
```

## 六、自动上传到 GitHub（核心一步）

发布命令（**先手动把 `spec.json` 的 `version` 改大**）：

```bash
node scripts/publish-spec.mjs --msg="chore(spec): 在这里写本次改了什么 v1.0.1"
```

脚本会依次做：

1. 解析并结构校验；
2. 校验版本号比上次大；
3. 把 `spec.json` 规范化为统一格式；
4. `git add spec.json` → `git commit` → `git push origin main`；
5. 调用 jsDelivr purge 接口刷新 CDN 缓存。

推送成功后，**用户下次打开 ID Aura 就会自动拉到新规范**，无需任何操作。

> 若只想先确认版本号有没有升对：`node scripts/publish-spec.mjs --version-check-only`

## 七、发布后怎么确认成功了

- **看设置页**：ID Aura「设置 → 规范」Tab 会显示当前版本号和来源徽章（显示 `remote` 表示已走线上源）。
- **直接看线上**：浏览器打开 `https://cdn.jsdelivr.net/gh/chengcheng067/Id-aura@main/spec.json`，确认 `version` 已更新。
- **急用户所急**：jsDelivr 一般几分钟内自动更新；若想立刻生效，发布脚本已自动调用 purge。手动 purge 地址：
  `https://purge.jsdelivr.net/gh/chengcheng067/Id-aura@main/spec.json`

## 八、安全与权限（重要，请团队遵守）

1. **不要把 GitHub 个人令牌（PAT）写进任何脚本或文档。** 本仓库脚本只调用 `git` / `gh`，凭据由你本机提供。
2. **检查远程地址是否含明文 token**：`git remote -v`。若看到 `https://<token>@github.com/...`，请去掉令牌：
   ```bash
   git remote set-url origin https://github.com/chengcheng067/Id-aura.git
   gh auth login        # 用 GitHub CLI 登录，之后 push 走系统凭据
   ```
3. **团队需要有该仓库的 push 权限**（让仓库管理员把你们加为 collaborator）。
4. 发布脚本产生的临时提交信息文件 `.spec-commit-msg` 会自动删除，无需关心。

## 九、常见问题（FAQ）

**Q：用户说没收到更新？**
A：最常见是 `version` 没升（软件只在版本变大时才更新）；其次检查用户网络，或让他重开一次软件。仍不行就手动 purge（见第七节）。

**Q：我改了规范，但离线/重装后还是旧的？**
A：那是 EXE 内置的离线兜底 `specs.bundled.js`，需走「方式 C」重新生成并打包发版。日常热更新不影响联网用户。

**Q：Excel 打开 CSV 中文乱码？**
A：导出时已带 UTF-8 BOM，正常 Excel 不会乱码。若仍乱码，用「数据 → 从文本/CSV」导入并选 UTF-8。

**Q：改错了怎么办？**
A：所有发布都走 git，可直接回滚：`git revert HEAD` 然后 `git push`；或 `git checkout <旧commit> -- spec.json` 后重新发布。

**Q：能不能不升版本号就发布？**
A：不能，也不建议。版本号是用户端判断「要不要更新」的唯一依据，不升用户永远看不到新内容。
