# PRD: AI API 配合设计规范库集成

> **项目**：ID Aura — 在线灵感图版 + 内嵌行业设计规范库工具  
> **版本**：v2.7.0  
> **日期**：2026-06-23  
> **状态**：已确认 · 可实施

---

## 1. 产品目标

**一句话产品目标**：让用户接入自己的 AI API（如 OpenAI / Claude / 国产大模型），将 ID Aura 内置的 30+ 品类设计规范数据作为上下文，通过对话式 AI 辅助设计师快速查询、校验和应用行业规范，降低查规范的时间成本，提高设计方案的合规率。

## 3. AI 接口与规范库配合的使用场景

### 场景 A：品类规范自动检查清单（AI 分析）

**触发方式**：用户在侧边栏选中某个品类（如"火锅店"）→ 点击"AI 分析"按钮

**流程**：
1. 前端读取该品类完整的 specs 数据（所有 sections + items）
2. 组装 system prompt：`你是一个餐饮工装设计规范顾问。以下是某品类的完整规范数据，请根据这些数据生成一份<检查清单>，按优先级（高/中/低）分类列出。`
3. 发送至用户配置的 AI API
4. 流式渲染 AI 返回的检查清单

**数据量示例**（火锅店）：
- 7 个 sections × 约 5-8 items = ~40-50 条规范
- 每条含 label/value/note/priority
- 序列化文本量约 3000-5000 tokens

### 场景 B：图片导入后 AI 自动识别空间并匹配规范

**触发方式**：用户拖入一张布局图/效果图到画布 → 右键菜单"AI 分析此空间"

**流程**：
1. 前端发送图片 base64 + 提示词（多模态模型支持时）
2. 用户可补充文字说明：如"这是一张 300㎡ 烧烤店的平面布局"
3. AI 识别空间类型 → 自动匹配对应品类规范 → 给出布局合规建议
4. 结果以卡片形式展示在对话面板

**限制说明**：
- 此场景仅**推荐支持视觉识别的模型**（如 GPT-4o / Claude 3.5 Sonnet / Gemini）
- 要求用户自行配置的 API 支持多模态输入
- 若不支持，降级为纯文本方案：用户手动选择品类，AI 仅基于规范数据做文本分析

### 场景 C：自定义 prompt + 规范数据上下文

**触发方式**：AI 对话面板中输入框 → 用户自由输入问题

**流程**：
1. 用户在对话输入框中键入问题
2. 前端将当前选中的品类 / 搜索中的规范数据作为上下文注入
3. 发送请求：`[用户问题] + [当前品类规范数据摘要]`
4. AI 给出基于规范的针对性回答
5. 支持多轮对话（保留历史消息）

**示例问答**：
- *Q*："我这间 200㎡ 的日料店，板前吧台区留多大合适？"
- *A*："根据日料店规范，板前吧台座建议占总座位数的 30-50%，每位宽度 600-700mm。200㎡ 约可设 30-40 座，其中板前座约 12-18 位..."

### 场景 D：跨品类规范对比

**触发方式**：用户选中多个品类 → "AI 对比分析"

**用例**：设计师在做复合业态（如"火锅+烧烤"混合店）时，让 AI 对比两个品类的规范差异，找出冲突点（如排烟量、防火等级不一致），给出兼容方案。

### 场景 E：布局图粗略合规校验

**触发方式**：用户在画布上摆放了若干规范卡片后 → 点击"AI 校验布局"

**用例**：设计师将火锅店的主通道、桌距、排烟等规范卡片拖到画布上，AI 根据画布上的规范条目，结合用户输入的"预估面积"，给出整体合规率评分和改进建议。

| ID | 需求 | 说明 |

## 5. 技术限制与边界

### 5.1 客户端纯前端方案（无后端）

> **决策确认**（2026-06-23）：先做纯前端方案（Web-first），Electron 版后续再增加 IPC 代理增强兼容性。

- AI 请求从浏览器**直接**发出 → 用户的 API Key 仅与 AI 服务商交互
- 优点：无服务端成本、数据出用户设备可控、部署简单
- 风险：**CORS 限制**（浏览器同源策略）
  - OpenAI API 默认支持 CORS，可直接调用
  - 其他 API 若不支持 CORS，需要用户自行配置 API 网关或使用浏览器插件
  - **建议方案**：在设置中提示用户开启"允许跨域"或使用支持 CORS 的 API 代理
  - **降级方案**：对于不支持 CORS 的 API，提示用户使用兼容层如 `one-api`、`new-api`

### 5.2 规范数据量评估

- 约 30 品类，每品类 ~5-8 sections，每 section ~3-8 items
- 每条 item 含 label / value / note / priority，序列化后约 100-200 字符
- **估算数据量**：
  - 单品类全文：约 1500-5000 tokens（取决于品类复杂度和 sections 数量）
  - 全部品类：约 50000-150000 tokens（超出多数模型上下文窗口，不适合全量注入）
  - **方案**：只注入当前选中的品类数据 + 用户搜索的关键词相关规范

### 5.3 Token 消耗估算

| 场景 | 输入 tokens（规范数据+用户消息） | 输出 tokens | 估算成本（GPT-4o-mini） |
|------|--------------------------------|------------|----------------------|
| 检查清单（火锅店） | ~4000 tokens | ~500 tokens | ~$0.001 |
| 自定义问答 | ~3500 tokens | ~300 tokens | ~$0.0008 |
| 图片分析（多模态） | ~20000 tokens（含图片） | ~400 tokens | ~$0.01 |

> 用户自行承担 API 费用，应用无需支付。

### 5.4 隐私考虑

| 内容 | 存储位置 | 说明 |
|------|---------|------|
| API Key | localStorage | 明文存储，不清除浏览器不会丢失 |
| 对话历史 | React state（内存） | 刷新页面后清空 |
| 规范数据 | 本地代码中硬编码 | 不发送至任何第三方 |
| 用户图片 | 仅发送至用户配置的 API | 不经过任何服务端中转 |

**安全提示**：应在设置界面提示用户"API Key 仅存储于本地浏览器，请妥善保管"。

### 5.5 请求失败/超时处理

| 失败类型 | 表现 | 提示文案 |
|---------|------|---------|
| 网络断开 | fetch 抛出 TypeError | "网络连接失败，请检查网络后重试" |
| 超时（>30s） | AbortController 终止 | "请求超时，请检查 API 服务状态或重试" |
| HTTP 401 | API Key 无效 | "API Key 无效，请在设置中检查" |
| HTTP 404 | Model Name 不存在 | "模型不存在，请检查模型名称是否正确" |
| HTTP 429 | 速率限制 | "请求过于频繁，请稍后重试" |
| CORS 错误 | 浏览器拦截 | "跨域请求被阻止，请确认 API 服务支持 CORS 或使用代理" |

### 5.6 CORS 限制说明

- OpenAI API (`https://api.openai.com/v1`) — ✅ 支持 CORS
- Anthropic API (`https://api.anthropic.com/v1`) — ✅ 支持 CORS
- 国内模型 API（百度文心、阿里通义、DeepSeek 等）— ⚠️ 部分支持，部分需用户自行配置
- 自建 API 网关（one-api / new-api）— ⚠️ 需要网关服务配置允许前端域名
- **建议**：在设置页中增加"API 兼容性说明"链接

### 6.3 AI 面板交互模式

**决策确认**（2026-06-23）：AI 面板内置两种交互模式，用户可在面板顶部切换：

| 模式 | 适用场景 | 说明 |
|------|---------|------|
| **快速问答** | 日常查规范、快速验证 | 单轮 + 多轮上下文，AI 基于规范数据做事实性回答，响应速度快 |
| **深度分析** | 方案审计、合规评分、多维度对比 | AI 进行多轮推理、思维链分析、对比分析、排版建议，适合需要深入分析的场景 |

- 两种模式共享同一对话历史
- 切换模式不影响已有对话记录
- 深度分析模式可在 system prompt 中额外追加推理指导指令

### 6.4 预设 Prompt 模板

- 初始预置模板：规范检查清单、规范白话解释、合规校验、跨品类对比、布局优化建议

## 7. 待确认问题（已确认·可实施）

> 以下 Q1-Q6 经用户审核确认，所有决策已锁定，可直接进入实施。

### Q1：用户期望的交互深度？— ✅ 已确认

**决策**：快速问答（A）和深度分析（B）**两种模式都内置**，让用户在 AI 面板中自行选择使用哪个模式。

- 快速问答：单轮对话 + 多轮上下文，AI 基于规范数据做事实性回答，适合"快速查规范"
- 深度分析：AI 进行多轮推理、思维链分析、对比分析、排版建议，适合"方案审计"
- 两种模式共存，在 AI 面板顶部提供切换开关
- 详见 §6.3 AI 面板交互模式

### Q2：是否需要预设 prompt 模板？— ✅ 已确认

**决策**：预置 3-5 个模板，但 UI 层级有明确约束——模板选择器**仅在用户唤出 AI 模块后**再显示，**不**平铺在主界面。

- 初始模板列表：规范检查清单、规范白话解释、合规校验、跨品类对比、布局优化建议
- 模板选择器位于 AI 对话面板内部（输入区附近或底部状态栏）
- 详见 §6.4 预设 Prompt 模板

### Q3：是否需要规范数据之外的专业知识库？— ✅ 已确认

**决策**：理论上需要外部专业知识库（地方性规范差异、最新法规更新等），但本次实现作为 **P2 / 远期处理**，不阻塞 Phase 1。

- Phase 1-3 仅基于内置 30 品类规范数据
- AI 回答约束：prompt 中注明"仅基于当前内置规范回答，超出部分请注明"
- 已追加 R18（外部专业知识库）至 P2 需求清单
- 详见 §4 需求清单 R18

### Q4：API 调用是否需要在主进程（Electron）中转发？— ✅ 已确认

**决策**：按 PRD 原建议执行——先做**纯前端方案（Web-first）**，Electron 后续再增加 IPC 代理增强兼容性。

- Phase 1 浏览器直接发请求
- 后端/Electron IPC 代理作为未来增强项
- 详见 §5.1 客户端纯前端方案（无后端）

### Q5：是否支持多模态模型？— ✅ 已确认

**决策**：支持多模态。在"测试连接"成功后，通过小请求**探测模型能力**，据此动态启用/禁用图片分析功能。

- 若模型支持多模态：启用图片上传/分析入口
- 若模型不支持：降级为纯文本方案（用户手动选择品类）
- 详见 §4 R04（连接测试含能力探测）及 §3 场景 B 限制说明

### Q6：对话历史长度限制？— ✅ 已确认

**决策**：按原建议执行——保留最近 **10-20 轮**（或约 **8000 tokens**），超过自动丢弃最早消息。用户可手动清空对话。

- 对话历史存储在 React state（内存），刷新页面清空
- 详见 §4 R07（多轮对话）及 §5.4 隐私考虑
